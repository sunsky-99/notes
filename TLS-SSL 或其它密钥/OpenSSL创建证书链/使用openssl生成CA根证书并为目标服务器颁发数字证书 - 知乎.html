本文用于记述如何使用openssl生成CA根证书并为目标服务器颁发数字证书。

假设有一个名叫xxoo的组织，它要自己弄个CA，给自己的内部服务颁发数字证书。

一、环境记述

  • 操作系统: linuxmint 20.1，与ubuntu类似。
  • openssl: 3.0.1 (该版本为本地自行编译安装，参考文章: linuxmint升级openssl )
  • 工作目录: 在任意目录下创建一个工作目录，如: ~/xxoo

1.1 查看openssl支持的椭圆曲线

openssl ecparam -list_curves

本文准备使用secp384r1曲线，它对应的是ecdsa签名算法的NIST P-384曲线。

你也可以选择其他椭圆曲线，如常用的NIST系列:

  • prime256v1 对应 NIST P-256
  • secp384r1 对应 NIST P-384
  • secp521r1 对应 NIST P-521
目前不推荐使用RSA算法，其满足相同安全强度下，需要2048位以上的密钥长度。

如果对数字证书，椭圆曲线等概念缺少必要的了解，请查看文章:

二、生成CA根证书

首先为xxoo生成自签名的CA根证书，其组织为xxoo，单位部门是ca，通用名为ca.xxoo.com。

2.1 创建相关目录

在工作目录下创建ca相关目录:

mkdir -p ~/xxoo/ca/certs
mkdir ~/xxoo/ca/keys
cd ~/xxoo

2.2 生成私钥文件

创建nistP384曲线私钥，作为CA的私钥:

openssl ecparam -genkey -name secp384r1 -out ca/keys/ca.key

2.3 创建CA根证书申请

创建证书申请，指定证书拥有者，即subject:

openssl req -new \
  -subj "/C=CN/ST=Anhui/L=Hefei/O=xxoo/OU=ca/CN=ca.xxoo.com" \
  -key ca/keys/ca.key -out ca/certs/ca.csr

# 查看CA证书申请
openssl req -text -noout -verify -in ca/certs/ca.csr

2.4 创建CA证书扩展信息

证书的扩展字段文件，在后面签署x509证书时，通过-extfile传入。

这里写入比较重要的几个扩展字段:

echo "subjectKeyIdentifier=hash" > ca/certs/ca_cert_extensions
echo "authorityKeyIdentifier=keyid:always,issuer" >> ca/certs/ca_cert_extensions
echo "basicConstraints=critical,CA:true" >> ca/certs/ca_cert_extensions
echo "subjectAltName=DNS.1:localhost,DNS.2:ca.xxoo.com,IP:127.0.0.1" >> ca/certs/ca_cert_extensions

说明:

  • subjectKeyIdentifier : 证书拥有者ID
  • authorityKeyIdentifier : 证书颁发者ID，对于自签名证书，它与subjectKeyIdentifier相同
  • basicConstraints : 基础约束，用于识别是否是CA证书
  • subjectAltName : SAN扩展信息，用于证书信任链构建时检查域名等信息

2.5 生成自签名的CA根证书

通过openssl x509命令签署一个自签名证书，即CA根证书。

openssl x509 -req -days 3650 -sha256 \
  -extfile ca/certs/ca_cert_extensions \
  -signkey ca/keys/ca.key \
  -in ca/certs/ca.csr  -out ca/certs/ca.crt

# 查看CA根证书
openssl x509 -text --noout -in ca/certs/ca.crt

注意openssl x509命令的参数-signkey传入的是CA自己的私钥。

三、为目标服务器颁发证书

模拟目标服务器的组织为xxoo，单位部门是server，通用名为server.xxoo.com。

通过以下操作生成server/keys/server.key与server/certs/server.crt，然后将它们发送给目标服务器即可。

3.1 创建相关目录

mkdir -p server/certs
mkdir server/keys

3.2 为目标服务器生成私钥

为目标服务器生成私钥，仍使用nistP384曲线:

openssl ecparam -genkey -name secp384r1 -out server/keys/server.key

3.3 创建证书请求

注意OU与CN的设置:

openssl req -new \
  -subj "/C=CN/ST=Anhui/L=Hefei/O=xxoo/OU=server/CN=server.xxoo.com" \
  -key server/keys/server.key -out server/certs/server.csr

openssl req -text -noout -verify -in server/certs/server.csr

3.4 生成服务器证书扩展信息

注意basicConstraints与CA证书不同，并添加了keyUsage字段:

echo "subjectKeyIdentifier = hash" > server/certs/server_cert_extensions
echo "authorityKeyIdentifier = keyid:always,issuer" >> server/certs/server_cert_extensions
echo "basicConstraints = CA:FALSE" >> server/certs/server_cert_extensions
echo "keyUsage = nonRepudiation, digitalSignature, keyEncipherment" >> server/certs/server_cert_extensions
echo "subjectAltName = DNS.1:localhost,DNS.2:server.xxoo.com,IP:127.0.0.1" >> server/certs/server_cert_extensions

3.5 生成服务器证书

这里也使用openssl x509命令进行签署，但是注意签署使用的私钥是CAkey:

openssl x509 -req -days 3650 -sha256 \
  -extfile server/certs/server_cert_extensions \
  -CA ca/certs/ca.crt -CAkey ca/keys/ca.key \
  -in server/certs/server.csr -out server/certs/server.crt

openssl x509 -text --noout -in server/certs/server.crt

注意:

  • -CA : 颁发者证书,即CA证书
  • -CAkey : 颁发者用于签署该证书的私钥，注意是ca的私钥。

四、验证

自行生成的CA根证书在没有加入本地或浏览器的信任证书列表时，直接访问会有警告，但我们可以将其加入信任证书列表，避免警告。

4.1 浏览器验证

假定一个https服务已经使用了server/keys/server.key私钥文件与server/certs/server.crt证书文件，并启动了https服务(端口443)，那么可以通过浏览器(如chrome)访问，查看其证书。

此时浏览器(如chrome)认为证书是不安全的，会有警告，我们可以在浏览器(如chrome)的设置中找到证书管理，在授权机构中导入前面生成的CA根证书ca/certs/ca.crt。

然后再重新访问https服务，就会发现证书已经是安全的了。

4.2 openssl验证

也可以使用openssl验证:

# 将CA根证书加入本地证书信任列表
sudo cp ca/certs/ca.crt /usr/local/share/ca-certificates/ca_xxoo_nistP384.crt
sudo update-ca-certificates
# 查看本地信任证书列表中是否有该CA根证书
ll /etc/ssl/certs | grep xxoo_nistP384

# 结果示例:
# 489b4616.0 -> ca_xxoo_nistP384.pem
# ca_xxoo_nistP384.pem -> /usr/local/share/ca-certificates/ca_xxoo_nistP384.crt*

# 执行ssl客户端命令访问
openssl s_client -connect <https服务Host>:<https服务端口> -CAfile /etc/ssl/certs/ca_xxoo_nistP384.pem -servername <https服务Host>
